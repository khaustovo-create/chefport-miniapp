# ShefPort — Product & Data Spec

## Описание продукта

ShefPort — Telegram Mini App для оформления заказов в розничном магазине. Покупатель просматривает каталог, собирает корзину и оформляет заказ через бота; администратор магазина обрабатывает заказы и при необходимости вручную правит каталог. Каталог, цены и остатки синхронизируются с **МойСклад** через API.

**MVP:** один бот, одно Mini App, одна точка продаж.  
**На будущее:** отдельный инстанс (бот + Mini App + МойСклад-аккаунт) на каждый магазин; в схеме данных заложен `shop_id` для миграции без переделки.

---

## Роли

| Роль | Кто | Права | Видимость |
|------|-----|-------|-----------|
| **Покупатель** | Telegram-пользователь | Каталог, корзина, оформление заказа, отмена заказа (только статус «новый»), просмотр своих заказов | Только свои заказы и корзина |
| **Администратор магазина** | Владелец / сотрудник | Просмотр всех заказов, смена статусов, отмена на любом этапе, ручное управление каталогом | Все заказы и каталог своей точки |
| **МойСклад** | Внешняя система (API) | Push/pull каталога, цен, остатков | Не пользовательская роль |

## Ключевые сценарии

### 1. Оформление заказа (покупатель)

1. Покупатель открывает Mini App в Telegram.
2. Листает каталог по категориям, добавляет товары в корзину.
3. Указывает имя и телефон, нажимает «Оформить».
4. Заказ создаётся в БД со статусом **новый**.
5. Администратор получает уведомление в Telegram.

### 2. Обработка заказа (администратор)

1. Администратор получает уведомление о новом заказе.
2. Открывает список заказов, меняет статус: **новый → подтверждён → готов → выдан**.
3. При каждой смене статуса покупатель получает уведомление.
4. Каждое изменение статуса записывается в историю.

### 3. Синхронизация каталога (система ↔ МойСклад)

1. По расписанию (остатки и цены — каждые 30 мин; полный каталог — раз в день) или вручную по запросу админа.
2. Система запрашивает товары, цены и остатки из МойСклад API.
3. Обновляет таблицу товаров в БД.
4. Покупатель видит актуальный каталог (с учётом `is_available` и `archived`).

### 4. История заказов (покупатель)

1. Покупатель открывает Mini App → раздел «Мои заказы».
2. Видит список заказов с датами, составом и текущими статусами.

## Lifecycle данных

| Сущность | Создание | Обновление | Удаление / архивация |
|----------|----------|------------|----------------------|
| **Пользователь** | Автоматически при первом входе в Mini App | Имя, username из Telegram; phone — при оформлении заказа | Не удаляется |
| **Товар** | Синхронизация из МойСклад или вручную админом | Цена, остаток, метаданные при sync; ручные правки админом | `is_available = false` при остатке 0; `archived = true` если товар пропал из МойСклад |
| **Корзина** | При добавлении первого товара | При каждом изменении корзины | Автоочистка через 7 дней без активности |
| **Заказ** | При нажатии «Оформить» | Смена статуса админом или покупателем (отмена) | Не удаляется, хранится вечно |
| **История статусов** | При каждой смене статуса заказа | — | Не удаляется |
| **Уведомление** | При отправке Telegram-уведомления | Обновление `delivery_status` при ошибке | Не удаляется (для дебага) |

## Multi-tenancy

| Аспект | MVP (сейчас) | Целевая модель |
|--------|--------------|----------------|
| Бот / Mini App | Один | Отдельный на каждый магазин |
| Каталог | Один | Свой на каждую точку |
| МойСклад | Один аккаунт | Свой аккаунт на точку |
| Администратор | Один | Свой на точку |
| Покупатель | Привязан к боту (= точке) | То же |
| Изоляция данных | `shop_id = 1` (константа) | Отдельные инстансы БД; `shop_id` в таблицах — задел на объединение |

## Аудит и история

| Что | MVP |
|-----|-----|
| История статусов заказа | **Обязательно** — когда, old → new, кто (покупатель / админ) |
| История изменений каталога | Не нужна |
| История цен и остатков | Не нужна |
| Лог действий админа | Не нужен |
| Факт отправки уведомлений | **Желательно** — кому, когда, тип, статус доставки (без текста) |

## Data

> Все таблицы содержат `shop_id` (default `1` на MVP).  
> Первичные ключи — UUID или BIGSERIAL; внешние ключи с `ON DELETE RESTRICT`, кроме явно указанных.

### 1. Shop

Точка продаж. На MVP — одна запись.

| Поле | Тип | Обяз. | Описание |
|------|-----|-------|----------|
| `id` | BIGINT PK | да | Идентификатор точки |
| `name` | VARCHAR(255) | да | Название магазина |
| `telegram_bot_token` | VARCHAR(255) | да | Токен бота (хранить зашифрованным) |
| `moysklad_account_id` | VARCHAR(64) | да | ID аккаунта МойСклад |
| `admin_telegram_ids` | BIGINT[] | да | Telegram ID администраторов |
| `created_at` | TIMESTAMPTZ | да | Дата создания |

---

### 2. User

Покупатель (Telegram-пользователь).

| Поле | Тип | Обяз. | Описание |
|------|-----|-------|----------|
| `id` | BIGINT PK | да | Внутренний ID |
| `shop_id` | BIGINT FK → Shop | да | Точка продаж |
| `telegram_id` | BIGINT | да | Telegram user ID |
| `first_name` | VARCHAR(255) | нет | Имя из Telegram |
| `last_name` | VARCHAR(255) | нет | Фамилия из Telegram |
| `username` | VARCHAR(255) | нет | @username |
| `phone` | VARCHAR(20) | нет | Телефон (заполняется при заказе) |
| `created_at` | TIMESTAMPTZ | да | Первый вход |
| `updated_at` | TIMESTAMPTZ | да | Последняя активность |

**Уникальность:** `(shop_id, telegram_id)`

### 4. Product

Товар каталога.

| Поле | Тип | Обяз. | Описание |
|------|-----|-------|----------|
| `id` | BIGINT PK | да | ID товара |
| `shop_id` | BIGINT FK → Shop | да | Точка продаж |
| `category_id` | BIGINT FK → Category | нет | Категория |
| `moysklad_id` | VARCHAR(64) | нет | ID в МойСклад |
| `name` | VARCHAR(500) | да | Название |
| `description` | TEXT | нет | Описание |
| `price` | DECIMAL(12,2) | да | Цена (руб.) |
| `stock` | INT | да | Остаток |
| `unit` | VARCHAR(32) | нет | Ед. измерения (шт, кг…) |
| `image_url` | VARCHAR(500) | нет | URL изображения |
| `is_available` | BOOLEAN | да | `false` при stock = 0 |
| `archived` | BOOLEAN | да | `true` если пропал из МойСклад |
| `is_manual_override` | BOOLEAN | да | Админ правил вручную (default false) |
| `synced_at` | TIMESTAMPTZ | нет | Последняя синхронизация |
| `created_at` | TIMESTAMPTZ | да | — |
| `updated_at` | TIMESTAMPTZ | да | — |

**Индексы:** `(shop_id, is_available, archived)`, `(shop_id, category_id)`, `(moysklad_id)`

**Правило видимости в каталоге:** `is_available = true AND archived = false`

### 5. Order + OrderItem

Заказ и его позиции. Цены в позициях — snapshot на момент оформления.

#### Order

| Поле | Тип | Обяз. | Описание |
|------|-----|-------|----------|
| `id` | BIGINT PK | да | ID заказа |
| `shop_id` | BIGINT FK → Shop | да | Точка продаж |
| `user_id` | BIGINT FK → User | да | Покупатель |
| `order_number` | VARCHAR(32) | да | Человекочитаемый номер |
| `status` | ENUM | да | `new`, `confirmed`, `ready`, `issued`, `cancelled` |
| `customer_name` | VARCHAR(255) | да | Имя при оформлении |
| `customer_phone` | VARCHAR(20) | да | Телефон при оформлении |
| `total_amount` | DECIMAL(12,2) | да | Сумма заказа |
| `cancelled_by` | ENUM | нет | `buyer`, `admin` — при status = cancelled |
| `created_at` | TIMESTAMPTZ | да | Дата оформления |
| `updated_at` | TIMESTAMPTZ | да | Последнее изменение |

**Индексы:** `(shop_id, status, created_at DESC)`, `(user_id, created_at DESC)`

#### OrderItem

| Поле | Тип | Обяз. | Описание |
|------|-----|-------|----------|
| `id` | BIGINT PK | да | ID позиции |
| `order_id` | BIGINT FK → Order | да | Заказ |
| `product_id` | BIGINT FK → Product | да | Товар (RESTRICT) |
| `product_name` | VARCHAR(500) | да | Snapshot названия |
| `unit_price` | DECIMAL(12,2) | да | Snapshot цены |
| `quantity` | INT | да | Количество |
| `line_total` | DECIMAL(12,2) | да | unit_price × quantity |

### 6. CartItem

Корзина покупателя. TTL — 7 дней без активности (по `updated_at` пользователя или позиции).

| Поле | Тип | Обяз. | Описание |
|------|-----|-------|----------|
| `id` | BIGINT PK | да | ID позиции корзины |
| `shop_id` | BIGINT FK → Shop | да | Точка продаж |
| `user_id` | BIGINT FK → User | да | Покупатель |
| `product_id` | BIGINT FK → Product | да | Товар |
| `quantity` | INT | да | Количество (> 0) |
| `created_at` | TIMESTAMPTZ | да | Добавлено |
| `updated_at` | TIMESTAMPTZ | да | Изменено |

**Уникальность:** `(user_id, product_id)`  
**Индекс:** `(user_id, updated_at)`

| Поле | Тип | Обяз. | Описание |

### 8. Notification

Журнал отправленных Telegram-уведомлений (для дебага).

| Поле | Тип | Обяз. | Описание |
|------|-----|-------|----------|
| `id` | BIGINT PK | да | ID |
| `shop_id` | BIGINT FK → Shop | да | Точка продаж |
| `recipient_telegram_id` | BIGINT | да | Кому отправлено |
| `type` | ENUM | да | `new_order`, `status_changed` |

| `delivery_status` | ENUM | да | `sent`, `failed` |

### Статусы заказа (ENUM)

```
new ──→ confirmed ──→ ready ──→ issued
 │          │            │           │
 └──────────┴────────────┴───────────┴──→ cancelled
```

| Переход | Кто может |
|---------|-----------|
| new → confirmed / ready / issued | Админ |
| new → cancelled | Покупатель или админ |
| confirmed / ready → cancelled | Только админ |
| issued | Финальный успешный статус |
