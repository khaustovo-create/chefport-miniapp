# PRODUCT_SCENARIOS.md — Поведение ШефПорта

ШефПорт — Telegram Mini App для покупателей рыбного магазина. Покупатель листает каталог и оформляет заказы прямо в боте; админ магазина получает уведомления, отслеживает остатки и статистику через команды бота. Каталог, цены и остатки синхронизируются с МойСклад через API в реальном времени.

---

## Сценарий 1: Покупатель открывает магазин и видит каталог

**Кто:** Покупатель (Telegram-пользователь)

**Триггер:**
Пользователь открывает чат с ботом ШефПорт и нажимает кнопку **«🛒 Открыть магазин»** (или кликает inline-кнопку).

**Нормальный поток:**

1. Минипрограмма загружает HTML и JavaScript со статического сервера
2. Фронтенд отправляет `POST /api/visit` с данными пользователя (id, username, first_name)
   - Визит логируется в SQLite таблицу `visits` (для статистики).
3. Фронтенд запрашивает `GET /api/products` → бэк вызывает `getFrontendProductsWithStock()`
4. Если кэш товаров свежий (< 60 сек):
   - Бэк возвращает закэшированный каталог → клиент получает ответ < 200 мс
5. Если кэш истёк:
   - Бэк запрашивает МойСклад:
     - `GET /report/stock/all` (остатки по складам, с пагинацией до 1000 строк за раз)
     - `GET /entity/product` и `/entity/variant` (детали товаров: категория, цена, описание)
     - `GET /entity/{type}/{id}/images` (для каждого товара с изображением, батчами по 5 + задержка 200 мс)
   - Собирает товары в массив `{ id, name, price, category, image, uomName, assortmentHref, assortmentType }`
   - Сортирует по категориям (А-Я по-русски)
   - Сохраняет в in-memory кэш с TTL 60 сек
6. Пользователь видит каталог:
   - Полный список товаров, сгруппированный по категориям
   - Фильтр "Все" / категории (переключается без перезагрузки)
   - Для каждого товара: название, цена, изображение (420px), единица измерения (шт/кг)

**Результат для пользователя:**
✓ Каталог загружен и показан на экране
✓ Товары отсортированы логично (по названиям внутри категорий)
✓ Изображения видны (или пропущены, если товар без фото)
✓ Цены в рублях (без копеек, целые числа)

**Edge cases:**
- **Если МойСклад недоступен (timeout, 500):**
  - Если есть старый кэш (>60 сек) → вернуть его (пользователь видит может быть устаревший каталог, но лучше чем ошибка)
  - Если кэша нет → вернуть 500 с ошибкой "МойСклад недоступен"
  
- **Если товар без изображения:**
  - Пропустить запрос на картинку, отдать товар с пустым `image: null`
  
- **Если много товаров (>250 SKU):**
  - Категории загружаются быстро (в памяти)
  - Но сам запрос к МойСклад может занять 2-3 сек при холодном кэше
  
- **Если категория товара содержит иерархию (папка/подпапка):**
  - Брать последний уровень ("Подпапка", а не "Папка/Подпапка")

**Guardrails:**
⚠️ **МойСклад API лимиты:** Батчи запросов изображений ограничены 5 одновременно с задержкой 200 мс между батчами. Иначе МойСклад вернёт 429 (Too Many Requests). Это закодировано в `IMG_CONCURRENCY = 5`, `IMG_BATCH_DELAY_MS = 200`.

⚠️ **Пагинация остатков:** `GET /report/stock/all` возвращает максимум 1000 строк. Если товаров больше → нужен offset-loop. Текущий код это учитывает в `getStockForWarehouseFast()`.

⚠️ **Фильтрация по складу:** Товары, отсутствующие на складе (stock = 0 или null), не показываются. Это фильтруется в `buildShowcase()` до формирования ответа.

⚠️ **Кэш сбрасывается при рестарте:** Это in-memory кэш. Если процесс упадёт — кэш потеряется, но первый запрос просто заново заполнит его.

⚠️ **Цены в копейках в МойСклад:** API возвращает цены в копейках (умножить на 100). Конверсия: `priceRub = Math.round(kopeks) / 100` — это гарантирует целый рубль, без погрешности.

## Сценарий 2: Покупатель добавляет товары в корзину и оформляет заказ

**Кто:** Покупатель (Telegram-пользователь)

**Триггер:**
Пользователь видит каталог, нажимает "+" на товаре, сильно его любит (добавляет много), потом нажимает кнопку **«✅ Оформить заказ»**.

**Нормальный поток:**

1. Пользователь добавляет товар в корзину (нажимает "+" или вводит количество):
   - `setQty(product, qty)` обновляет Map корзины (id → {name, price, qty, uomName, assortmentHref, assortmentType})
   - Внизу экрана обновляется строка "🛒 N товаров • M ₽" (количество + сумма)
   
2. Пользователь видит кнопку "Оформить" → нажимает
   
3. Открывается form с полями:
   - Имя (текстовое поле)
   - Телефон (маска +7(XXX) XXX-XX-XX, обязательное, для поиска в /buyer)
   - Адрес доставки (опциональное)
   - Комментарий (опциональное)
   
4. Пользователь заполняет форму и нажимает **«Отправить заказ»**
   
5. Фронтенд отправляет `POST /api/order` с телом:
   ```json
   {
     "items": [
       { "id", "name", "price", "qty", "uomName", "assortmentHref", "assortmentType" },
       ...
     ],
     "customer": { "name": "...", "phone": "...", "address": "..." },
     "comment": "...",
     "user_id": "123456789"
   }
   ```

6. Бэк обрабатывает заказ:
   - Проверяет, что корзина не пуста и все позиции валидны
   - Для каждого товара формирует позицию для МойСклад:
     - Конвертирует цену в копейки: `priceKopeks = Math.round(priceRub * 100)`
     - Использует `assortmentHref` и `assortmentType` из фронтенда (это ссылки на Product или Variant в МойСклад)
   - Ищет или создаёт контрагента (покупателя) по номеру телефона:
     - Если найден → использует существующего
     - Если нет → создаёт нового контрагента в МойСклад
   - Создаёт заказ (`CustomerOrder`) в МойСклад с этими позициями и описанием (имя, телефон, адрес, комментарий)
   - Возвращает 200 с `{ ok: true, order: { id, name, ... } }`

7. Бэк отправляет уведомление админу в Telegram:
   - Fire-and-forget, не блокирует ответ клиенту
   - Сообщение содержит: имя, телефон, адрес, состав корзины (позиции с qty × price), номер заказа в МойСклад
   - Если админ не сконфигурирован (нет `TELEGRAM_ADMIN_CHAT_ID`) → уведомление тихо пропускается

8. Бэк логирует заказ в SQLite таблицу `orders`:
   - `ms_order_name`: номер заказа из МойСклад
   - `user_id`: Telegram user ID
   - `customer_name`, `customer_phone`: из формы
   - `total_sum`: сумма заказа (в рублях)
   - `items_json`: JSON-массив позиций
   - `created_at`: текущая дата/время UTC

9. Пользователь видит сообщение **«✅ Заказ принят!»** и корзина очищается

**Результат для пользователя:**
✓ Заказ создан в МойСклад и отмечен уникальным номером (e.g., "Заказ MiniApp 2026-06-23 14:30:45")
✓ Админ получил уведомление в телеграме с полной информацией о заказе
✓ Корзина очищена, пользователь может начать новый заказ
✓ Статистика обновлена (новая строка в таблице `orders`)

**Edge cases:**
- **Если контрагент с таким телефоном уже существует в МойСклад:**
  - Используется существующий контрагент (его данные могут быть другие — система не обновляет профиль, только привязывает новый заказ)
  
- **Если форма не полностью заполнена:**
  - Если пустые имя или телефон → фронтенд показывает ошибку, кнопку «Отправить» блокирует
  - Если телефон неправильного формата (не цифры) → фронтенд не отправляет запрос
  - Адрес и комментарий опциональны, их можно оставить пусто
  
- **Если МойСклад недоступен при создании заказа:**
  - Бэк возвращает 500 с ошибкой "МойСклад недоступен"
  - Заказ НЕ логируется в БД (логирование в try/catch, но только если заказ в МойСклад создан)
  - Пользователь видит ошибку и может повторить попытку (корзина не очищена)
  
- **Если цена в корзине давно загруженная и между загрузкой и оформлением цена изменилась:**
  - Бэк отправляет в МойСклад цену из корзины (которую пользователь видел и согласился)
  - МойСклад создаёт заказ с этой ценой, даже если цена в каталоге другая
  - Админ видит в МойСклад реальную цену, может заметить расхождение, вручную отредактировать

- **Если в корзине товар с количеством в кг (не целое число, e.g. 1.5 кг):**
  - Это поддерживается, quantity может быть дробное число
  - МойСклад примет такое количество

**Guardrails:**
⚠️ **assortmentHref должен быть корректный:** Если фронтенд отправляет неправильный `assortmentHref` → заказ в МойСклад создастся, но не привяжется к правильному товару. Эту ошибку надо ловить на фронтенде. Правильный href берётся из ответа `/api/products`.

⚠️ **Хрупкость с бот.sendMessage():** В коде `bot.sendMessage()` вызывается до объявления переменной `bot` ниже в файле. Это работает только потому, что вызов асинхронный, и к тому времени, когда `sendMessage()` попробует отправить, переменная `bot` уже инициализирована. Это fragile и может сломаться, если изменить порядок инициализации.

⚠️ **Цены в копейках:** При создании заказа цены конвертируются: `Math.round(priceRub * 100)`. Если поправлять цены в будущем → всегда помнить про эту конверсию.

⚠️ **Телефон используется как уникальный ключ контрагента:** Если один и тот же человек закажет с разными телефонами → создадутся разные контрагенты. Если человек захочет обновить телефон → нужно делать это вручную в МойСклад.

⚠️ **Уведомление админу не блокирует ответ:** Если отправка уведомления упадёт → клиент всё равно получит 200 OK. Заказ создан, но админ может не узнать. Это попадает в конкурирующий race condition если админ одновременно запрашивает /stats или /buyer.

## Сценарий 3: Админ проверяет статистику магазина

**Кто:** Администратор магазина (сотрудник или владелец)

**Триггер:**
Админ пишет команду `/stats` в чате с ботом.

**Нормальный поток:**

1. Админ отправляет сообщение `/stats`

2. Бот проверяет, что отправитель — это админ:
   - Если `msg.chat.id === TELEGRAM_ADMIN_CHAT_ID` → продолжить
   - Иначе → молча проигнорировать команду (без ошибки)

3. Бот запрашивает из SQLite таблиц `visits` и `orders`:
   - Три периода: Сегодня, Неделя (6 дней назад), Месяц
   - Для каждого периода считает:
     - **Визиты:** total (сколько раз кто-то открыл Mini App) и unique_users (сколько разных people)
     - **Заказы:** total количество и revenue (сумма в рублях)

4. Бот запрашивает топ 5 товаров по количеству проданных за месяц (из `items_json` в таблице `orders`):
   - Парсит JSON каждого заказа, суммирует qty по товарам, сортирует по убыванию
   - Для каждого товара: название, total qty, unit (шт/кг)

5. Бот запрашивает количество повторных покупателей за месяц:
   - Это пользователи, которые сделали заказ более одного раза за месяц
   - SELECT DISTINCT `customer_phone` WHERE created_at >= date('now', 'start of month') HAVING COUNT(*) > 1

6. Бот отправляет админу красиво отформатированное сообщение:
   ```
   📊 Статистика магазина
   
   📅 Сегодня
     Визиты: 42 (18 уник.)
     Заказы: 5 на 3 850 ₽
   
   📅 Неделя
     Визиты: 287 (95 уник.)
     Заказы: 35 на 24 120 ₽
   
   📅 Месяц
     Визиты: 1200 (340 уник.)
     Заказы: 142 на 98 540 ₽
   
   🏆 Топ товаров (месяц)
     1. Стерлядь филе — 45 кг
     2. Лосось слайс — 38 кг
     3. Кальмар кольца — 27 шт
     4. Креветка очищенная — 15 кг
     5. Крабовое мясо — 8 банок
   
   🔁 Повторных покупателей (месяц): 24
   ```

**Результат для пользователя (админа):**
✓ Админ видит трёхуровневую статистику (день/неделя/месяц)
✓ Понимает, какие товары в тренде
✓ Видит ratio уникальных визитов к заказам (конверсия)
✓ Может планировать закупки на основе спроса

**Edge cases:**
- **Если за период нет ни одного визита или заказа:**
  - Показывать "0" и "0 ₽" (не hide, просто ноль)
  
- **Если админ ещё не сконфигурирован:**
  - Команда тихо игнорируется, админ не получит никакого ответа
  
- **Если таблицы `visits` или `orders` пусты:**
  - Вернуть 0 по всем метрикам (не ошибка)
  
- **Если товар в топе повторяется (одинаковое имя в разных заказах):**
  - Просто суммировать qty по названию (не умная дедупликация по ID)

- **Если товар без unit (uomName):**
  - Показывать как "шт" по умолчанию

**Guardrails:**
⚠️ **Admin guard:** Команда `/stats` сравнивает `String(msg.chat.id) !== String(TELEGRAM_ADMIN_CHAT_ID)`. Оба значения преобразуются в строки, потому что Telegram ID может быть число или строка в разных контекстах.

⚠️ **WAL-режим SQLite:** Таблица `visits` и `orders` используют WAL (Write-Ahead Logging), что безопасно для конкурентных SELECT-запросов от разных подпроцессов, но если бэк одновременно логирует новый заказ и админ смотрит `/stats` → есть tiny race condition (админ может увидеть результат до коммита). Это нормально для MVP.

⚠️ **Парсинг JSON items:** Если `items_json` в какой-то строке невалидный JSON → это поймается в try/catch и значение пропустится (не сломает весь топ). Это хорошо, но логирует молча.

⚠️ **Временные зоны:** Все даты в БД хранятся в UTC (`strftime('%Y-%m-%dT%H:%M:%SZ', 'now')`). Фильтры по периодам используют функции SQLite `date('now')` и `date('now', 'start of month')`. Если сервер работает не в UTC → может быть сдвиг на N часов. Это задача для глобального проекта, пока можно игнорировать.

⚠️ **Топ товаров считается по названию, не по ID:** Если один товар записан как "Креветка очищенная" и "Креветка (очищенная)" → будут two separate entries в топе. Это дефект, но низкий приоритет.

## Сценарий 4: Админ ищет все заказы покупателя по номеру телефона

**Кто:** Администратор магазина

**Триггер:**
Админ получил звонок или СМС от покупателя "А где мой заказ?" Админ пишет `/buyer +79991234567` (или любой формат телефона).

**Нормальный поток:**

1. Админ отправляет сообщение `/buyer <телефон>`
   - Примеры: `/buyer +7 999 123 45 67`, `/buyer 79991234567`, `/buyer +7(999)123-45-67`

2. Бот парсит аргумент после команды, trim() его

3. Бот проверяет админские права:
   - Если `msg.chat.id !== TELEGRAM_ADMIN_CHAT_ID` → молча проигнорировать

4. Бот запрашивает из SQLite:
   ```sql
   SELECT * FROM orders WHERE customer_phone = ? ORDER BY created_at DESC
   ```
   - Используется индекс `idx_orders_phone` для быстрого поиска

5. Если заказов не найдено:
   - Ответить "Заказов по номеру +7(999)123-45-67 не найдено."

6. Если заказы найдены:
   - Отформатировать список:
   ```
   📋 История заказов: +7(999)123-45-67

   • 2026-06-20  Заказ MiniApp 2026-06-20 14:30:45  2 500 ₽
     Имя: Иван Петров
     — Стерлядь филе × 1.5
     — Лосось слайс × 0.5
   
   • 2026-06-15  Заказ MiniApp 2026-06-15 09:15:22  1 850 ₽
     Имя: Иван Петров
     — Креветка очищенная × 1
   ```

**Результат для пользователя (админа):**
✓ Админ видит всю историю покупок этого клиента
✓ Видит даты и суммы заказов
✓ Видит состав каждого заказа (позиции и количество)
✓ Может понять, постоянный ли это клиент, и какой средний чек

**Edge cases:**
- **Если формат телефона разный:**
  - Бот ищет в БД точное совпадение того, что админ указал в /buyer
  - Если админ пишет "/buyer 9991234567", а в БД записано "+79991234567" → ноль результатов
  - Админ должен использовать ту же запись номера, что и пользователь при оформлении

- **Если в одном заказе много позиций:**
  - Все они перечисляются в сообщении (может быть длинное сообщение)
  - Telegram режет на 4096 символов, но для MVP маловероятно

- **Если items_json невалидный JSON:**
  - Попробовать распарсить, если ошибка → пропустить позиции и показать только сумму

- **Если заказов больше 20:**
  - Показываются только последние 20 (отсортированы по дате, новые первыми)
  - Это лимит в коде `getOrdersByPhone()` LIMIT 20 для быстрого ответа и вписывания в Telegram

**Guardrails:**
⚠️ **Точное совпадение телефона:** Нет нормализации номера. Если клиент один раз ввёл "+79991234567", а в другой раз "9991234567" → это будут разные записи в БД. Надо вручную инструктировать админов использовать единообразный формат.

⚠️ **Производительность при больших объёмах:** Если один номер телефона заказывал 1000 раз (маловероятно для MVP), запрос SELECT вернёт все 1000 строк, Telegram сообщение упадёт. Для MVP приемлемо, для масштаба нужна пагинация.

⚠️ **Admin guard:** Та же проверка как в `/stats`, с преобразованием String.

⚠️ **Конфиденциальность:** Команда `/buyer` раскрывает все заказы по любому номеру. Если админ не один — может быть утечка. Для MVP предполагаем, что админ один и доверяем ему. На продакшене нужна аудит.

⚠️ **LIMIT 20 в getOrdersByPhone():** Запрос в db.js возвращает максимум 20 последних заказов. Это лимит закодирован для безопасности (Telegram режет сообщения на 4096 символов). Если клиент имел 50+ заказов, админ увидит только 20 новых.

## Сценарий 5: Система синхронизирует каталог с МойСклад

**Кто:** Фоновый процесс (через запросы фронтенда)

**Триггер:**
- Пользователь открывает Mini App (срабатывает GET /api/products)
- Прошло 60+ секунд с момента последнего обновления каталога (кэш истёк)

**Нормальный поток:**

1. Фронтенд запрашивает `GET /api/products`

2. Бэк проверяет in-memory кэш в `moysklad.js`:
   - Если `(now - cache.ts) < 60000` миллисекунд → вернуть закэшированные товары (быстро, < 50 мс)
   - Иначе → запросить свежие данные (см. ниже)

3. Запросить из МойСклад:
   - `GET /report/stock/all?warehouse=WH_ID` (offset-loop, до 1000 строк за раз)
     - Это даёт текущие остатки, папку товара, цену, ID ассортимента (product/variant)
     - Фильтруем: берём только строки, где stock > 0
   
   - Для каждого товара запросить `GET /entity/{type}/{id}` с expand:
     - expand=images (получим images.meta с count, но не сами downloadHref)
     - expand=productFolder (категория товара)
     - expand=uom (единица измерения)
     - expand=salePrices.priceType (цены по типам)
   
   - Для каждого товара с images.meta.size > 0 запросить `GET /entity/{type}/{id}/images`
     - Получим реальный downloadHref для картинки
     - Батчи: 5 товаров параллельно, потом 200 мс пауза, следующий батч
   
   - Собрать структуру для фронтенда:
     ```json
     {
       "products": [
         {
           "id": "uuid",
           "name": "Стерлядь филе",
           "price": 580,
           "category": "Рыба",
           "image": "/api/image?src=...",
           "uomName": "кг",
           "assortmentHref": "https://api.moysklad.ru/...",
           "assortmentType": "product"
         },
         ...
       ],
       "categories": ["Рыба", "Морепродукты", ...],
       "updatedAt": "2026-06-23T14:35:00Z"
     }
    ```
   
   - Сохранить в кэш с TTL 60 сек

4. Вернуть бэку 200 JSON с каталогом

5. Фронтенд рендерит каталог на UI

**Результат для пользователя:**
✓ Каталог всегда актуальный (максимум 60 сек отставания)
✓ Цены и остатки из МойСклад в реальном времени
✓ Быстрая загрузка (200 мс из кэша, 2-3 сек на холодный)
✓ Если МойСклад упал → старый кэш спасает (graceful degradation)

**Edge cases:**
- **Если между открытием каталога и оформлением заказа цена изменилась:**
  - Фронтенд хранит цену в корзине (цена в момент добавления)
  - При оформлении отправляет эту цену в МойСклад
  - МойСклад создаст заказ с ценой из фронтенда (не обновляет на текущую)
  - Админ может заметить расхождение, отредактировать цену вручную в МойСклад

- **Если товар добавлен в МойСклад после холодного запуска (товар нет в кэше):**
  - Ждём 60 сек, потом кэш обновится
  - Товар появится в каталоге на следующем запросе

- **Если товар удалён из МойСклад:**
  - На следующее обновление товар пропадёт из каталога (не будет в report/stock/all или stock = 0)
  - Если товар был в корзине у пользователя → при оформлении заказа это может быть ошибкой (товара нет в МойСклад, ссылка неправильная)

- **Если изображение товара больше 8 MB:**
  - Запрос на `/api/image` упадёт с 504 (timeout или maxContentLength exceeded)
  - Фронтенд покажет broken image или заглушку

- **Если конкурентно открывают много пользователей (холодный кэш):**
  - Первый запрос будет slow (2-3 сек)
  - Остальные посчитают, что кэш в процессе обновления, и дождутся или упадут с timeout?
  - Текущий код не синхронизирует параллельные запросы (race condition), но маловероятна для одного процесса

**Guardrails:**
⚠️ **Батчи изображений:** Максимум 5 одновременных запросов, 200 мс между батчами. Если это изменить → МойСклад вернёт 429. Константы в `moysklad.js`: `IMG_CONCURRENCY = 5`, `IMG_BATCH_DELAY_MS = 200`.

⚠️ **Пагинация остатков:** `limit=1000` за запрос. Если в складе >1000 позиций → нужен offset-loop. Код это делает, но надо помнить при масштабировании.

⚠️ **Цены в копейках:** МойСклад возвращает цены в копейках × 100. Конверсия: `Math.round(kopeks) / 100`. Всегда юзать эту функцию, иначе погрешности в рубли.

⚠️ **In-memory кэш теряется при рестарте:** Если процесс упадёт и перезагрузится → первый запрос будет медленный (холодный запрос к МойСклад). Это нормально для MVP, но для продакшена можно добавить Redis или персистентный кэш.

⚠️ **Фильтр по складу:** Все запросы фильтруются по `MOYSKLAD_WAREHOUSE_ID`. Если этот ID неправильный или изменился → каталог будет пуст. Это надо проверить при запуске (в log).

⚠️ **extractAssortmentMetaFromRow() критична:** Эта функция извлекает ID, тип и href товара. Если логика сломается → заказ не привяжется к товару. Менять только с тестами.
